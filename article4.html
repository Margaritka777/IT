Всесвітня павутина сьогодні стала надто складною. З сотнями й тисячами електронних засобів масової інформації, які спілкуються між собою, процес спілкування став надто складним і напевно важким для обчислень. Такі сценарії вимагають чогось, що може полегшити спілкування та звести до мінімуму складність кодування. Це щось є предметом цієї публікації та називається REST.

REST, або RE presentational S State T Transfer, є архітектурним стилем і забезпечує стандарти між системами в Інтернеті. Зазвичай системи RESTful використовують стиль REST для спілкування одна з одною. Тож у багатьох місцях ви можете побачити в Інтернеті «RESTful» , і це означає те саме. У цій статті ми детальніше зупинимося на основах REST і розглянемо наступні теми в цій статті.

Що таке REST – репрезентативна передача стану?

REST ( RE presentational  S tete  Transfer ) вперше був представлений у 2000 році Роєм Філдінгом як архітектурний стиль для розподілених гіпермедійних систем. REST-сумісні або RESTful системи є «без стану» (розглянуто далі в цій статті) і розділяють клієнта та сервер. Веб-програма, розроблена за допомогою REST (веб-програма RESTful), надає дані або інформацію про свої ресурси, які можуть бути будь-якими, що забажає розробник. Це дозволяє клієнту, який використовує цю програму, виконувати дії з ресурсами. Наприклад, використовуючи інформацію, надану користувачам, клієнти можуть створити нового користувача.

Як інший приклад, припустімо, що у нас є зоомагазин, і інформація про всіх домашніх тварин у магазині зберігається на сервері. Якщо нам потрібна інформація, пов’язана з домашньою твариною (скажімо, домашня тварина з ідентифікатором = 0), тоді ми отримаємо доступ до відповідної URL-адреси зоомагазину в браузері (тут браузер буде клієнтом). Прикладом URL-адреси може бути "https://demoqa.com/BookStore/v1/Books".

Представлення потоку REST

Тепер давайте зобразимо фактичну передачу даних REST у наведеному вище прикладі на діаграмі нижче.

Як показано на діаграмі вище, API працює як засіб зв’язку між базою даних (тобто частиною сервера) та клієнтом. Коли клієнт надсилає дані через API, він переходить до бази даних, виконує відповідну операцію (наприклад, додає, видаляє, змінює тощо) і повертає дані відповіді, які містять код відповіді, файли заголовків, інформацію про файли cookie тощо.

Ми можемо узагальнити характеристики REST і роботу на наведеній вище діаграмі таким чином:

  • Дані доступу клієнта з сервера передані REST. Це може бути або через авторизований доступ, або без дотримання будь-яких суворих вказівок.
  • Як ми бачимо, додаток, розроблений за допомогою REST, є інтерактивним додатком і в основному використовує веб-сервіси. У цьому випадку веб-служба відповідає вказівкам RESTful і виконує обмеження REST, які обговорюються далі в цій статті.
  • Веб-служба, що використовує REST, надає веб-ресурси в текстовому форматі та дозволяє їх читати та редагувати за допомогою попередньо визначеного набору операцій.

REST — це спосіб доступу до веб-сервісу, який часто розглядається як альтернатива SOAP (простий протокол доступу до об’єктів).

У додатку RESTFul у нас є сутності, а саме клієнт і ресурс, які використовуються зазвичай. Коротко обговоримо їх далі.

Що таке клієнти та ресурси?

Клієнтом може бути програмне забезпечення, особа або система, яка використовує API для доступу до даних з іншого сервера додатків. Наприклад, розробник може отримати доступ до API Facebook, щоб вставити живу публікацію на власний веб-сайт. Програма розробника викличе Facebook API через браузер. Тож у цьому випадку браузер діє як клієнт, який викликає API Facebook.

Якщо ми візуалізуємо цю систему за допомогою діаграми REST вище, клієнт або браузер підключиться до Facebook-сервера через REST API, а потім отримає інформацію, необхідну для відтворення на екрані.

Як інший приклад, припустімо, що у мене є програма "myHealthApp". Мені потрібні дані про пандемію Covid-19 з міста, наприклад Пуни. Щоб досягти цього, я розроблю API або методи, як-от «Corona API», за допомогою яких я можу запитувати дані від муніципальної ради Пуни ( PMC ). Це робиться за допомогою myHealthApp. Це означає, що використовуючи myHealthApp, який діє як клієнт, я буду робити запит на дані за допомогою Corona API (скажімо, метод getCoronaData, щоб бути більш конкретним) через браузер (наприклад, https://myHealthApp/getCoronaData ). Цей метод getCoronaData у свою чергу підключатиметься до серверів PMC і отримуватиме необхідні дані у вигляді відповіді JSON (або будь-якого іншого формату). На стороні клієнта я можу взяти ці дані та виконувати з ними різні операції. Це використання API.

Що таке ресурс у REST?

Будь-який менший блок, який можна трансформувати та адресувати через URL-адресу та метод HTTP, вважається ресурсом. Цей ресурс вносить зміни в базу даних. Для однієї програми у вас може бути багато ресурсів, усім із яких призначено певне завдання. Наприклад, книжковий інтернет-магазин може мати ресурс у вигляді таблиці бази даних. Ресурс в архітектурі REST — це все, до чого клієнт має доступ і може змінювати чи оновлювати.

У певному сенсі можна сказати, що ресурс – це будь-який об’єкт, для якого нам потрібна інформація з API. Отже, у наведеному вище прикладі про Facebook ресурс може бути публікацією, сторінкою або обліковим записом користувача. У прикладі Corona API ресурси — це всі деталі щодо Corona, як-от дані Corona, сторінки про лікування, вакцинацію тощо.

Керівні принципи або обмеження REST

Щоб API відповідав REST, він має виконувати або дотримуватися наведених нижче керівних принципів або обмежень, визначених REST.

Уніфікований інтерфейс

Принцип уніфікованого інтерфейсу включає наступні частини, яким має відповідати API:

  1. Запит до сервера повинен мати ідентифікатор ресурсу.
  2. Сервер повертає відповідь і містить таку інформацію, що клієнт може змінити або відредагувати ресурс.
  3. Запит, надісланий через API, містить інформацію, необхідну серверу для виконання запиту. Кожна відповідь, яку повертає сервер, також містить усю інформацію, щоб клієнт зрозумів відповідь.
  4. Гіпермедіа як двигун стану додатків. Програма означає веб-програму, яка працює на сервері. Гіпермедіа — це гіперпосилання або прості посилання, включені у відповідь. Таким чином, сервер в основному інформує клієнта про способи зміни веб-програми.

Розділення клієнт-сервер

Взаємодія між клієнтом і сервером є незалежною і здійснюється лише у формі запитів. Клієнт ініціює запит, а сервер надсилає відповіді. Відповідь є реакцією на запит. Тому все, що робить сервер, це чекає отримання запитів від клієнта. Він ніколи не надсилає інформацію про ресурси клієнту самостійно. Щоб отримати докладніші відомості про клієнт-сервер, зверніться до нашого підручника « Клієнт-серверна архітектура».

Без громадянства

Слово «без стану» означає, що сервер не відстежує користувачів, які використовують API. Отже, коли надходить новий запит, сервер не знатиме, чи той самий користувач уже надсилав запит GET для того самого ресурсу. Він не запам'ятовує ресурси, запитувані користувачем раніше. Наприклад, HTTP є протоколом без стану. Сервер HTTP не відстежує будь-який стан інформації, що передається туди-сюди. Таким чином, у будь-який момент часу клієнт може надіслати дійсну команду, а сервер не зв’язуватиме та не відстежуватиме попередні подібні команди.

Отже, кожен запит, незалежно від інших запитів, зроблених користувачем, міститиме всі дані, необхідні серверу для виконання запиту та надсилання відповіді.

Пошарова система

Багатошарова система забезпечує ієрархічну структуру між клієнтом і сервером. Між клієнтом і сервером може бути багато посередників, які працюють разом із REST API без повідомлення клієнта. Наші клієнти вважають, що є пряме підключення до сервера. Потім ми використовуємо це, щоб покращити нашу архітектуру та знизити складність розподіленої системи. Ці проміжні елементи забезпечують рівень безпеки, рівень балансування навантаження та інші функціональні можливості системи. Єдина вказівка ​​полягає в тому, що наявність цих проміжних рівнів не повинна впливати на запит або відповідь.

Примітка: абстракція шарів не дозволяє одному шару знати про наявність іншого шару.

Кешується

Дані сервера, отримані у відповіді, містять інформацію про те, кешуються дані чи ні. Якщо дані можна кешувати, вони міститимуть певний номер версії, який уможливлює кешування. Клієнт знатиме, яку версію даних він отримав із попередньої відповіді. Таким чином клієнт може уникнути повторного запиту даних. Дані, які можна кешувати (і, отже, номер версії) також допомагають клієнту дізнатися про закінчення терміну дії даних версії та необхідність нового запиту для отримання останніх даних.

Код на вимогу

Це конкретне обмеження «Код на вимогу» є необов’язковим, і без його виконання ми можемо мати RESTful API.

Клієнт може надіслати серверу запит на код, а потім сервер відповість деяким кодом у формі сценарію або іншої сутності. Наприклад, сервери можуть розширювати функціональність клієнта, завантажуючи та виконуючи попередньо скомпільований код, як-от аплет, або сценарій на стороні клієнта, як-от JavaScript. Отже, коли ми натискаємо будь-яке відео на Facebook, Facebook запускає попередньо скомпільоване або будь-яке стороннє програмне забезпечення для запуску цього відео.

Якщо API відповідає вищезазначеним обмеженням, ми можемо сказати, що це RESTful API.

Ключові висновки

У цій статті ми обговорили запровадження протоколу REST для веб-служб.

  1. REST (REpresentational State Transfer) — це стандарт, який працює між машинами, які переважно виконують веб-сервіси.
  2. Сервер отримує запит у формі REST API. Потім сервер запускає веб-додаток і надсилає відповідь клієнту.
  3. Щоб API був дійсним REST API (RESTful API), він має відповідати деяким обмеженням, які ми обговорювали вище.
  4. RESTful API дотримується цих обмежень (їх також називають керівними принципами).
